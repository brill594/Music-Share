# Music Share 路线图

## 项目目标

做一个从 Android 手机端一键分享当前播放歌曲的轻量系统：

- 从 Poweramp 获取当前播放曲目
- 将源音频快速转码为适合上传和试听的压缩格式
- 上传到自建后端并生成短时可访问的分享链接
- 通过 Web 播放页展示歌曲信息和在线试听能力

## 关键前提与技术约束

在正式实现前，先固定几个不会绕开的事实：

1. Poweramp API 中的 `path` 不能再默认视为“文件系统绝对路径”。
   Android 10 及以上场景下，Poweramp 返回的 `path` 更接近“相对于已授权 SAF tree 的路径标识”，例如 `primary/Music/Album/song.flac`，而不一定是可直接访问的 `/storage/emulated/0/...`。

2. 目标 App 不能继承 Poweramp 的存储权限。
   即使 Poweramp 已经能访问某个目录树，当前项目的 Android App 仍然必须自己通过 SAF 获得对应目录的持久授权。

3. 正确做法不是拼接绝对路径，而是解析为当前 App 自己可访问的 `content://` URI。
   推荐做法是：
   - 保存 Poweramp 提供的原始 `path`
   - 让用户首次授权音乐目录的 `treeUri`
   - 基于 `treeUri + path` 生成对应文件的 `documentUri`
   - 后续所有读文件、提取封面、转码都围绕 `content://` URI 完成

4. 移动端本地状态至少要保存三类数据：
   - 当前曲目的 Poweramp 原始 `path`
   - 当前曲目的已解析 `content://` URI
   - 当前曲目的基础元数据快照，如标题、艺术家、专辑、时长、更新时间

## 阶段一：移动端 App（Android 核心入口）

这个模块是整个项目的基石。目标不是做复杂 UI，而是把“无感获取、极速转码、一键上传”这条链路跑通并稳定下来。

### 1. 状态监听（Poweramp API 接入）

实现目标：

- 接收 Poweramp 播放状态变更
- 提取当前曲目标识
- 将“可分享的当前曲目状态”落到本地轻量存储

实现路线：

1. 使用 `BroadcastReceiver` 接入 Poweramp 广播。
2. 重点监听：
   - `com.maxmpz.audioplayer.TRACK_CHANGED`
   - `com.maxmpz.audioplayer.STATUS_CHANGED`
3. 兼容两种数据来源：
   - 旧格式：从 `track` bundle 中取字段
   - 新格式：从 `STATUS_CHANGED` 顶层 extras 取平铺后的 track 字段
4. 从广播中提取：
   - `path`
   - `title`
   - `artist`
   - `album`
   - `duration`
   - `track id` 或其他稳定标识
   - 最近更新时间戳
5. 首次启动时引导用户通过 `ACTION_OPEN_DOCUMENT_TREE` 授权音乐目录，并调用 `takePersistableUriPermission` 保存权限。
6. 将 Poweramp 的 `path` 映射为当前 App 可访问的 `documentUri`：
   - 不依赖 `/storage/emulated/0/...` 绝对路径
   - 优先按 `treeUri` 解析为 `DocumentsContract` 目标 URI
7. 把最新曲目状态写入本地轻量数据库，优先选：
   - Room
   - 或极简场景下使用 DataStore + 单记录缓存

建议数据模型：

- `poweramp_path`: Poweramp 原始路径字符串
- `document_uri`: 当前 App 可访问的 `content://` URI
- `title`
- `artist`
- `album`
- `duration_ms`
- `art_uri` 或封面缓存路径
- `updated_at`
- `is_resolvable`: 当前路径是否成功映射到可访问 URI

验收标准：

- 切歌后 1 秒内可更新本地状态
- App 被系统回收后，重新收到广播仍能恢复记录
- Android 10+ 下不依赖绝对路径也能定位到真实文件

### 2. 触发机制（Quick Settings Tile）

实现目标：

- 用户从状态栏下拉菜单点击一次即可发起分享

实现路线：

1. 实现 `TileService`，在 Quick Settings 中提供“分享音乐”入口。
2. 点击 Tile 后执行以下编排：
   - 唤醒后台处理入口
   - 读取本地最新曲目记录
   - 校验 `documentUri` 是否仍可访问
   - 若不可访问，给出重新授权提示
   - 若可访问，进入转码与上传流程
3. Tile 的状态设计为极简三态：
   - Idle
   - Processing
   - Failed / Need Permission

建议实现方式：

- Tile 只负责触发，不做重逻辑
- 真正流程放在 `ForegroundService` 或 `WorkManager` 中执行
- 如果转码和上传可能超过几秒，优先使用前台服务，避免被系统杀掉

验收标准：

- 屏幕锁定或 App 未打开时，Tile 仍可正常发起处理
- 处理期间不会因后台限制被系统中断

### 3. 音频处理与元数据提取（FFmpeg 集成）

实现目标：

- 在手机端快速生成适合分享的轻量试听版本
- 保留必要元数据
- 允许用户按网络环境调整输出格式和压缩参数

实现路线：

1. 集成 `ffmpeg-kit-audio` 或同级别轻量音频方案。
2. 读取源文件时优先基于 `content://` URI 打开输入流。
3. 转码输出应做成可配置项，由用户在设置中选择：
   - 输出格式
   - 码率
   - 采样率
   - 声道数
   - 位深策略
4. 默认输出格式为 `OGG`
   - 默认编码优先 `Opus`
   - 若兼容性或设备环境受限，可回退 `Vorbis`
5. 可选格式应限制在“体积较小、适合小带宽传输”的范围内，建议首版只提供：
   - `OGG Opus`
   - `OGG Vorbis`
   - `AAC / M4A`
   - `MP3`
6. 不建议首版提供：
   - `FLAC`
   - `WAV`
   - `AIFF`
   - 其他无损或高体积格式
7. 默认转码档位：
   - 默认码率 `96 kbps`
   - 可选 `64 / 96 / 128 kbps`
   - 默认采样率 `44100 Hz`
   - 可选 `24000 / 32000 / 44100 / 48000 Hz`
   - 默认声道为 `2`
   - 可选 `1 / 2`
8. 关于位深：
   - 对有损输出格式来说，不以“最终文件位深”作为核心用户配置
   - 若源文件需要重采样或重编码，可在内部统一到 `16-bit` PCM 工作流后再编码
   - UI 上可展示为“位深策略”或“处理精度”，但不建议让用户直接面对过多底层术语
9. 处理常见源格式：
   - FLAC
   - WAV
   - MP3
   - AAC / M4A
10. 元数据提取策略：
   - 尽量保留标题、艺术家、专辑、时长
   - 若封面可提取，则单独保存为缩略图并一起上传
11. 对大文件设置硬性限制：
   - 时长上限
   - 输出文件大小上限
   - 单次任务超时

建议配置项：

- `output_format`
- `audio_codec`
- `bitrate_kbps`
- `sample_rate_hz`
- `channels`
- `loudness_mode` 或是否做音量标准化
- `max_duration_seconds`
- `max_output_size_mb`

推荐预设：

- `Fast Share`
  - `OGG Opus`
  - `64 kbps`
  - `24000 Hz`
  - 单声道或双声道
- `Balanced`
  - `OGG Opus`
  - `96 kbps`
  - `44100 Hz`
  - 双声道
- `Better Quality`
  - `OGG Opus` 或 `AAC`
  - `128 kbps`
  - `44100 Hz`
  - 双声道

建议执行顺序：

1. 校验文件可读
2. 提取基础元数据
3. 抽取封面
4. 执行转码
5. 校验输出文件大小和时长
6. 进入上传

验收标准：

- 3 到 5 分钟常规歌曲可在可接受时间内完成转码
- 输出文件可被主流浏览器正常播放
- 元数据与原曲基本一致
- 用户可以切换不同压缩预设，兼顾音质和上传速度

### 4. 上传与反馈机制

实现目标：

- 将试听文件和元数据可靠上传
- 生成分享链接并立即反馈给用户
- 支持客户端回查分享状态和提前结束分享
- 允许用户自定义后端地址和鉴权方式
- 允许客户端执行可选的管理员身份验证

实现路线：

1. 使用 `Retrofit` 或 `OkHttp` 封装上传请求。
2. App 首次安装后自动生成一个安装级持久标识 `client_install_id`：
   - 使用高强度随机源生成
   - 长度应明显高于普通 UUID，可采用 32 到 64 字节随机值再编码为字符串
   - 保存在本地安全持久存储中
   - App 卸载后允许重建，不追求跨重装一致
3. 后续每次请求后端时都附带该标识，建议放在请求头中，例如：
   - `X-Client-Install-Id: <opaque-random-id>`
4. App 设置中应允许用户配置后端连接信息：
   - `base_url`
   - `port`
   - `auth_mode`
   - `basic_auth_password`
5. `port` 配置项紧跟在 `base_url` 后面，默认值为 `2087`。
6. 鉴权方式默认值为 `none`，可选值至少包括：
   - `none`
   - `basic`
7. 当鉴权方式为 `basic` 时：
   - 仅要求输入密码，不要求输入用户名
   - 客户端启动时通过统一认证入口发起一次 `basic auth` 风格认证
   - 后端根据该密码自动鉴别当前身份是普通权限还是管理员权限
   - 认证成功后下发短期 `cookie` 或 `api key`
   - 后续请求优先携带短期凭证，不再每次携带明文密码
   - 导出 JSON 时应明确告知该配置包含敏感信息
   - 密码输入框默认隐藏，提供“点击显示密码”切换
8. 认证入口应允许用户填写“普通访问密码”或“管理员密码”：
   - 两者都只需要密码
   - 都通过同一个认证流程提交
   - 后端自动识别角色并返回对应权限级别
9. 客户端应增加本地请求日志，用于记录认证行为和凭证状态：
   - 最近一次认证请求时间
   - 认证成功时间
   - 返回的 `expires_at`
   - 当前凭证类型
   - 当前权限角色
10. 每次打开 App 时，客户端先根据本地请求日志判断当前 `cookie` 或 `api key` 是否过期：
   - 若日志显示凭证仍在有效期内，则直接复用，不再请求后端认证
   - 若日志缺失、已过期或接近过期，则再发起认证请求
11. 当需要重新认证时：
   - 优先使用本地持久化保存的密码完成认证
   - 成功后更新请求日志并缓存新的 `cookie` 或 `api key`
   - 默认有效期约 1 天
12. 认证成功后：
   - 普通权限客户端可访问普通分享服务
   - 管理员权限客户端可访问后端全部服务，包括管理页
13. 实际请求地址应由客户端根据配置动态拼接：
   - `request_base = {base_url}:{port}`
   - 认证请求、上传、状态查询、终止分享请求都基于该地址发起
14. 上传内容包括：
   - 转码后的 OGG 文件
   - 标题、艺术家、专辑、时长
   - 封面图
   - 原始 Poweramp `path` 的脱敏版本或内部追踪 ID
   - 期望分享有效时长
15. 服务端返回：
   - 内部 `uuid`
   - 对外 `share_code`
   - 可分享 URL
   - 过期时间
   - 当前分享状态
16. 客户端拿到分享链接后：
   - 自动写入系统剪贴板
   - Toast 提示“已复制链接”
   - 视需要补充系统通知
17. 客户端应提供“我的分享”或等价轻入口，用于：
   - 查询当前分享是否已过期
   - 查询距离过期还剩多久
   - 主动提前结束分享
18. 客户端还应提供“后端管理”页面：
   - 仅管理员可见
   - 普通用户或未验证用户不可见
   - 可查看后端当前已有音乐
   - 可提前结束任意已有分享
19. 对失败场景分类处理：
   - 无当前曲目
   - 未授权目录
   - 文件不可读
   - 转码失败
   - 上传失败
   - 客户端标识缺失或无效
   - 后端地址无效
   - 鉴权失败
   - 短期凭证过期

验收标准：

- 分享完成后用户无需再手动复制链接
- 失败时用户能知道卡在“授权、读取、转码、上传”的哪一步
- 用户能从客户端看到分享是否仍有效，并可主动结束分享
- 用户可修改后端地址和鉴权方式，并在重启后保持生效
- 用户只有在获得管理员权限短期凭证后才能看到后端管理页
- 每次打开 App 会先根据请求日志判断短期凭证是否过期
- 未过期时不会重复向后端请求认证
- 获得短期凭证后后续请求不需要重复携带密码

管理员页面建议模块：

- `Music List`
  - 查看当前后端仍存在的音乐
  - 按 `active / expired / terminated` 过滤
- `Track Actions`
  - 手动提前结束指定分享

密码交互要求：

- 所有密码输入框默认隐藏内容
- 提供显式的“显示密码 / 隐藏密码”切换按钮
- 至少覆盖：
  - `basic_auth_password`
  - 管理员密码
- 密码显示状态只作为当前界面临时状态，不需要单独导出

### 5. 配置管理、备份与导入

实现目标：

- 允许用户导出当前 App 关键配置
- 允许在新设备或重装后导入配置
- 支持导出和导入 `client_install_id`
- 所有用户设置都必须本地持久化保存
- 所有持久化设置都应随 `client_install_id` 一并导出到 JSON 文件
- 包括后端地址和鉴权配置
- 包括管理员验证相关配置与状态缓存策略
- 包括认证请求日志与凭证过期信息

实现路线：

1. 在设置页增加“导出配置”和“导入配置”入口。
2. 配置文件格式统一为 JSON。
3. 所有分享相关、转码相关、UI 相关设置都必须持久化到本地配置存储中，建议使用：
   - `DataStore` 保存结构化设置
   - 安全存储或受控持久层保存 `client_install_id`
   - 安全存储或受控持久层保存敏感鉴权信息
4. `basic_auth_password` 与管理员验证密码都必须本地持久化保存。
5. 导出时从本地持久化配置中读取完整设置快照，而不是从内存临时状态拼装。
6. 导出时使用系统文档选择器，让用户决定保存位置。
7. 导入时使用系统文件选择器读取 JSON 文件。
8. 导入前执行结构校验、字段校验和版本校验。
9. 若导入文件中包含 `client_install_id`，允许用户选择：
   - 覆盖当前安装标识
   - 保留当前安装标识，仅导入其他参数
10. 若覆盖 `client_install_id`，客户端需明确提示：
   - 该操作会影响“我的分享”管理视图
   - 导入后客户端将以新的安装身份与后端交互
11. 导入完成后，立即刷新本地配置缓存和请求头注入逻辑，并将新设置持久化落盘。

建议 JSON 结构：

```json
{
  "schema_version": 1,
  "exported_at": "2026-04-04T20:00:00Z",
  "client_install_id": "opaque-random-install-id",
  "share_defaults": {
    "expire_after_seconds": 86400
  },
  "server": {
    "base_url": "https://music.example.com",
    "port": 2087,
    "auth_mode": "none",
    "basic_auth_password": ""
  },
  "admin": {
    "enabled": false,
    "password": ""
  },
  "session": {
    "auth_type": "cookie",
    "access_key": "",
    "expires_at": ""
  },
  "auth_log": {
    "last_auth_requested_at": "",
    "last_auth_succeeded_at": "",
    "last_role": "user",
    "last_auth_type": "cookie",
    "last_expires_at": ""
  },
  "transcode": {
    "output_format": "ogg",
    "audio_codec": "opus",
    "bitrate_kbps": 96,
    "sample_rate_hz": 44100,
    "channels": 2
  },
  "ui": {
    "show_share_status": true
  }
}
```

建议纳入导出的配置字段：

- `schema_version`
- `client_install_id`
- 默认分享有效时长
- 后端 `base_url`
- 后端 `port`
- 鉴权方式
- `basic_auth_password`
- 管理员开关
- 管理员密码
- 短期凭证缓存信息
- 认证请求日志
- 默认转码格式
- 默认码率
- 默认采样率
- 默认声道数
- 其他所有已持久化的分享设置
- 其他所有已持久化的转码设置
- 其他非敏感 UI 配置

不建议导出的内容：

- 临时分享历史缓存
- 当前播放态缓存
- 运行时日志
- 服务端返回的瞬时状态

敏感信息提醒：

- 若导出内容包含 `basic_auth_password`，导出前应明确提示用户该文件含敏感信息
- 若导出内容包含管理员密码，同样应明确提示该文件含敏感信息
- `basic_auth_password` 与管理员密码默认都应随配置文件一并导出

导入校验建议：

1. 必须校验 `schema_version`
2. 必须校验 JSON 结构完整性
3. 必须校验 `client_install_id` 长度和字符合法性
4. 对未知字段采取忽略而不是报错，保证后续兼容性
5. 对无效格式配置回退到安全默认值
6. 必须校验 `base_url` 格式是否合法
7. 必须校验 `port` 是否为合法端口号
8. 必须校验 `auth_mode` 是否为受支持枚举值
9. 当 `auth_mode = basic` 时，需校验密码字段格式
10. 若包含管理员密码配置，需校验其字段格式
11. 若包含短期凭证缓存，需校验过期时间和字段结构
12. 若包含认证请求日志，需校验时间字段和角色字段结构

持久化要求：

1. 设置修改后应立即写入本地持久化存储，而不是仅停留在内存中。
2. App 重启后必须完整恢复上次保存的设置。
3. 导出的 JSON 必须来自当前持久化配置快照。
4. 导入成功后，新的 JSON 内容必须完整覆盖本地持久化配置，或按用户选择保留当前 `client_install_id` 后再落盘。
5. 后端地址和鉴权方式属于核心连接配置，必须持久化。
6. 后端端口属于核心连接配置，必须持久化。
7. 管理员密码配置若用户填写，也必须持久化。
8. `basic_auth_password` 与管理员密码都必须本地持久化。
9. `basic_auth_password` 与管理员密码都必须随 JSON 配置一起导出。
10. 短期 `cookie` 或 `api key` 可持久化缓存，但过期后必须自动重新申请。
11. 认证请求日志必须本地持久化，用于启动时判断是否需要重新认证。

验收标准：

- 用户能导出 JSON 配置文件
- 用户能导入 JSON 配置文件
- `client_install_id` 可随配置一起导出和恢复
- 用户修改的设置在 App 重启后仍然存在
- 导出的 JSON 与本地已持久化设置保持一致
- 用户可配置后端地址，默认鉴权方式为 `none`
- 用户可配置后端端口，默认值为 `2087`
- 用户可切换到密码式 `basic auth` 认证
- 用户可分别保存普通访问密码和管理员密码
- 客户端会根据请求日志决定是否复用现有短期凭证
- 凭证未过期时不会重复请求认证
- 管理员权限由后端根据密码自动判定
- 所有密码默认隐藏，并支持点击显示密码
- `basic_auth_password` 与管理员密码都能持久化并随 JSON 导出
- 非法配置文件不会导致 App 崩溃

### 6. 阶段一建议里程碑

M1：监听和解析打通

- 能稳定拿到 Poweramp 当前曲目
- 能把 `path` 解析为 `documentUri`

- 能从 `documentUri` 成功读取音频
- 能产出可试听的 OGG 文件

M3：一键分享闭环

- Tile 点击后完成转码、上传、复制链接全流程

M4：配置与迁移可用

- 能导出 JSON 配置
- 能导入 JSON 配置
- 能恢复 `client_install_id` 和转码默认参数

## 阶段二：后端与存储（API 与生命周期管理）

后端目标是极简、稳定、可自动回收，只承担三件事：

- 临时接收上传后的音频与封面文件
- 对外提供统一的元数据查询和音频转发入口
- 在设定时间后自动销毁全部相关资源

后端不负责：

- 长期存档
- 用户系统
- 播放列表
- 搜索
- 推荐
- 转码

转码全部放在 Android 端完成，后端只做临时文件存储和转发。

### 1. 基础 API 构建

技术选型建议：

- FastAPI：更适合当前需求，开发速度快，足够支撑轻量中转场景
- Gin：可作为后续替代选项，但首版没有必要优先上 Go

推荐优先级：

- 先用 FastAPI 跑通完整闭环
- 首版先避免引入 Redis、对象存储、消息队列等额外组件

推荐职责模型：

1. App 上传一个已经转码完成的音频文件。
2. 每次请求都附带安装级持久标识 `client_install_id`。
3. 后端将文件保存到本地临时目录。
4. App 可同时提交期望的分享有效时长。
5. 后端保存一条极简元数据记录，并将其归属到对应 `client_install_id`。
6. 后端自动生成对外分享 URL，其路径为随机 16 位字符串。
7. Web 端通过该分享标识查询元数据。
8. App 可基于 `client_install_id` 查询自己创建的分享状态，或提前结束分享。
9. 后端启动时随机生成高熵密码，用于普通权限和管理员权限认证。
10. 认证流程统一通过 `basic auth` 风格提交密码，不要求用户名。
11. 后端根据密码自动判断当前身份级别。
12. 认证成功后后端下发短期 `cookie` 或 `api key`，默认有效期约 1 天。
13. 客户端后续请求优先携带该短期凭证，而不是重复发送密码。
14. 客户端可依据本地认证请求日志判断该短期凭证是否仍有效。
15. 管理员权限客户端可访问后端全部服务并查看管理页。
16. Web 播放器通过统一流接口读取音频。
17. 到期后删除文件和记录。

核心接口：

1. `POST /upload`
   - 接收已经转码完成的 OGG 音频文件和基础元数据
   - 接收客户端指定的分享有效时长
   - 要求附带安装级 `client_install_id`
   - 生成内部 UUID 和对外 16 位分享路径
   - 将音频、封面、元数据写入临时存储
   - 返回分享页 URL、过期时间和分享标识
2. `GET /track/{share_code}`
   - 返回前端渲染所需的歌曲元数据 JSON
   - 不直接暴露内部磁盘路径
3. `GET /stream/{share_code}`
   - 由应用层完成鉴权和状态校验
   - 再通过 Nginx `X-Accel-Redirect` 返回受控文件内容
   - 主要用于前端页面首次加载时一次性完整拉取音频
4. `GET /cover/{share_code}`
   - 如果存在封面文件则返回封面内容
   - 如果不存在则返回 404，由前端降级处理
5. `GET /client/shares`
   - 基于 `client_install_id` 返回当前客户端创建的分享列表和状态
6. `GET /client/shares/{share_code}`
   - 返回指定分享的管理态信息，如剩余有效期和当前状态
7. `POST /client/shares/{share_code}/terminate`
   - 提前结束分享
   - 结束后公开访问应立即失效
8. `POST /auth/login`
   - 使用密码进行统一认证
   - 后端自动判断是普通权限还是管理员权限
   - 验证成功后返回短期 `cookie` 或 `api key`
9. `GET /admin/tracks`
   - 仅管理员可访问
   - 返回后端当前已有音乐列表
10. `POST /admin/tracks/{share_code}/terminate`
   - 仅管理员可访问
   - 提前结束任意分享

推荐接口行为：

#### `POST /upload`

请求内容：

- `file`: 转码后的 OGG 文件
- `cover`: 可选封面图
- `title`
- `artist`
- `album`
- `duration_ms`
- `audio_mime`
- `client_created_at`
- `expire_after_seconds` 或 `expire_at`

请求头建议：

- `X-Client-Install-Id`

有效期策略建议：

1. 优先允许客户端上传 `expire_after_seconds`。
2. 后端根据服务端当前时间计算 `expires_at`。
3. 若客户端直接传 `expire_at`，后端必须重新校验而不是盲信。
4. 后端应设置允许范围，例如：
   - 最短 10 分钟
   - 最长 72 小时
5. 若客户端未传有效期，则使用默认值 24 小时。
6. 若超出允许范围，则后端截断到服务端允许值或直接返回 400。

处理步骤：

1. 校验 `X-Client-Install-Id` 是否存在且符合长度要求
2. 校验 MIME、文件大小、基础元数据长度
3. 生成 `uuid`
4. 生成随机 16 位对外分享标识 `share_code`
5. 以服务端 `base_url` 拼接生成分享 URL：
   - `share_url = {base_url}/{share_code}`
   - 若服务部署在非默认端口，可按公开访问策略决定是否拼入端口
6. 计算最终 `expires_at`
7. 创建临时存储目录，如 `storage/<uuid>/`
8. 写入：
   - `audio.ogg`
   - `cover.jpg` 或 `cover.webp`
   - `meta.json` 或数据库记录
9. 返回：
   - `uuid`
   - `share_code`
   - `share_url`
   - `track_url`
   - `stream_url`
   - `cover_url`
   - `expires_at`
   - `status`

#### 管理员验证机制

设计目标：

- 认证流程统一
- 只需要密码，不需要用户名
- 后端自动识别普通权限和管理员权限
- 认证成功后签发短期凭证，减少明文密码重复传输

实现建议：

1. 后端启动时随机生成高熵密码。
2. 可至少区分两类密码：
   - 普通权限密码
   - 管理员密码
3. 这两类密码都只通过统一认证入口提交，不要求用户名。
4. 后端根据提交的密码自动识别角色。
5. 认证成功后返回短期 `cookie` 或 `api key`。
6. 凭证默认有效期约 1 天。
7. 客户端每次启动时先检查本地认证请求日志中的过期时间。
8. 若日志表明凭证未过期，则直接复用现有凭证。
9. 若日志表明凭证已过期或日志无效，再执行重新认证。
10. 后续业务请求只携带短期凭证。

注意：

- 普通密码和管理员密码都属于同一套认证体系
- 客户端不需要维护用户名
- 明文密码不应在每个业务请求中重复出现
- 认证请求日志只用于判断是否需要重新认证，不应用作权限来源本身

#### `POST /auth/login`

请求内容：

- `password`

返回字段建议：

- `role`
- `auth_type`
- `session_key`
- `expires_at`

建议行为：

- 验证成功返回短时 `cookie` 或 `api key`
- 验证失败返回 401
- 返回的 `role` 至少包括：
  - `user`
  - `admin`
- 客户端后续请求统一使用该短期凭证
- 返回体应包含明确的 `expires_at`，供客户端写入请求日志

#### `GET /admin/tracks`

职责边界：

- 仅管理员可访问
- 返回当前后端仍存在的分享音乐列表
- 支持按状态筛选：
  - `active`
  - `expired`
  - `terminated`

返回字段建议：

- `share_code`
- `title`
- `artist`
- `created_at`
- `expires_at`
- `status`
- `client_install_id` 的脱敏摘要

#### `POST /admin/tracks/{share_code}/terminate`

职责边界：

- 仅管理员可访问
- 可提前结束任意分享
- 结束后公开访问立即失效

#### `GET /client/shares`

职责边界：

- 从请求头读取 `X-Client-Install-Id`
- 仅返回该客户端自己创建的分享
- 用于客户端展示“我的分享”列表

返回字段建议：

- `share_code`
- `share_url`
- `title`
- `artist`
- `created_at`
- `expires_at`
- `status`

`status` 建议值：

- `active`
- `expired`
- `terminated`
- `missing`

#### `GET /client/shares/{share_code}`

职责边界：

- 验证该分享是否归属于当前 `client_install_id`
- 返回该分享的管理态信息

返回字段建议：

- `share_code`
- `status`
- `created_at`
- `expires_at`
- `terminated_at`
- `remaining_seconds`

#### `POST /client/shares/{share_code}/terminate`

职责边界：

- 验证该分享是否归属于当前 `client_install_id`
- 将分享状态标记为 `terminated`
- 使公开访问立即返回失效状态
- 可同步触发文件删除，或交给后台清理任务尽快处理

推荐行为：

- 成功后 `GET /track/{share_code}` 返回 `410`
- 成功后 `GET /stream/{share_code}` 返回 `410`
- 终止操作应幂等，重复调用仍返回成功态

`share_code` 生成规则建议：

- 长度固定为 16
- 字符集使用大小写字母加数字，或仅用大小写字母与数字中的安全子集
- 必须使用高质量随机源生成
- 必须做唯一性检测，避免碰撞

推荐示例：

- `https://music.example.com/Ab3dE91kLm2Pq8Rs`

#### `GET /track/{share_code}`

返回字段建议：

- `uuid`
- `share_code`
- `title`
- `artist`
- `album`
- `duration_ms`
- `audio_mime`
- `expires_at`
- `cover_url`
- `stream_url`

返回原则：

- 只返回页面展示所需字段
- 不返回服务端文件系统路径
- 不返回移动端本地文件信息

#### `GET /stream/{share_code}`

职责边界：

- 根据 `share_code` 找到对应内部记录和临时音频文件
- 校验分享是否存在、未过期、未终止
- 不由应用进程直接长时间回传大文件
- 由应用返回 `X-Accel-Redirect` 头，交给 Nginx 提供内部文件访问
- 默认按“整文件下载”场景服务前端页面
- 设置正确的 `Content-Type`
- 可不把 `Range` 作为首版强依赖，因为前端会在加载时完整获取音频

这里的“转发”本质上是：

- 前端不直接访问磁盘文件
- 所有音频访问都通过后端统一出口完成
- 应用层负责权限和状态判断，Nginx 负责高效文件发送
- 便于控制过期、日志和清理
- 前端只需请求一次音频文件，后续播放、暂停、拖动都在本地完成

推荐 `X-Accel-Redirect` 方案：

1. 应用将实际音频文件存放在 Nginx 不直接暴露的内部目录。
2. `GET /stream/{share_code}` 命中后，应用校验状态。
3. 校验通过后返回：
   - `X-Accel-Redirect: /_protected_audio/<uuid>/audio.ogg`
   - `Content-Type`
   - `Content-Disposition`（如需要）
4. Nginx 通过 `internal` location 读取真实文件并返回给客户端。

好处：

- 降低应用进程传输大文件的开销
- 更适合将整文件下载压力交给 Nginx
- 更容易统一管理链接有效性和文件访问路径

#### `GET /cover/{share_code}`

职责边界：

- 若封面存在，则按图片类型输出
- 若不存在，则返回 404
- 不在后端动态裁剪或加工图片，避免引入额外开销

接口数据建议：

- `uuid`
- `share_code`
- `title`
- `artist`
- `album`
- `duration_ms`
- `audio_mime`
- `expires_at`
- `cover_url`
- `stream_url`

### 2. 存储设计（仅做临时文件存储）

后端只保留短期可访问所需的最小数据。

推荐落盘结构：

- `storage/<uuid>/audio.ogg`
- `storage/<uuid>/cover.jpg`
- `storage/<uuid>/meta.json`

或使用：

- `storage/<uuid>/audio.ogg`
- `storage/<uuid>/cover.jpg`
- SQLite 中保存元数据

推荐 MVP 做法：

- 音频文件和封面图直接存本地磁盘
- 元数据使用 SQLite
- 每条记录只保留最少字段

推荐元数据字段：

- `uuid`
- `share_code`
- `client_install_id`
- `title`
- `artist`
- `album`
- `duration_ms`
- `audio_mime`
- `audio_path`
- `cover_path`
- `created_at`
- `expires_at`
- `terminated_at`
- `status`

这里的 `audio_path` 和 `cover_path` 仅在服务端内部使用，绝不能返回给客户端。

这里建议将“内部主键”和“外部分享路径”分离：

- `uuid`：服务端内部存储主键，用于目录和内部追踪
- `share_code`：对外公开的 16 位随机字符串，用于 URL 路径和公开查询

这样做的好处：

- URL 更短，更适合分享
- 避免直接暴露内部存储标识
- 后续更容易替换底层存储实现

关于 `client_install_id`：

- 这是客户端管理自己分享记录的私有凭证之一
- 不应出现在公开分享 URL 中
- 不应返回给前端播放页
- 应视为高熵、不可猜的安装级随机标识

### 3. 限期销毁机制（核心）

这是后端设计的硬要求，不是附加功能。

实现路线：

1. 所有上传记录都写入 SQLite，并记录 `created_at` 和 `expires_at`。
2. 文件落盘时按 UUID 隔离目录，便于整目录删除。
3. 若用户主动结束分享，记录 `terminated_at` 并立即将状态改为 `terminated`。
4. 使用一个简单定时任务扫描过期或已终止记录。
5. 每次扫描按批删除：
   - 音频文件
   - 封面图
   - 整个 UUID 目录
   - 对应数据库记录
6. 若删除文件失败，记录日志并在下一轮继续重试。

默认策略建议：

- 默认存活 24 小时
- 服务端响应中明确返回过期时间
- 前端在页面上展示“链接将在何时失效”
- 最终有效期以服务端确认结果为准，不以客户端请求值为准
- 主动结束分享后，公开访问必须立即失效

不推荐首版引入：

- Redis TTL
- 对象存储
- CDN 回源
- 异步任务队列

这些能力对当前“临时文件存储和转发”目标都不是必需品。

### 4. 安全与资源限制

虽然后端很轻，但仍要防止被滥用。

建议最少加上以下限制：

1. 限制单文件大小，例如 15 MB 到 30 MB。
2. 限制上传 MIME，仅允许预期音频格式和图片格式。
3. 限制标题、艺术家、专辑字段长度。
4. 限制单 IP 或单设备的上传频率。
5. 不允许列目录、不暴露静态文件真实路径。
6. 所有文件访问都通过分享标识查询，不支持路径型访问。
7. 对外只暴露 `share_code`，不暴露内部 `uuid`。
8. `client_install_id` 只允许通过请求头传输，不写入公开 URL。
9. 管理接口必须校验 `share_code` 与 `client_install_id` 的归属关系。

如需最小鉴权，可在 App 上传时附加一个固定服务端密钥或签名头，避免接口被随意灌数据。

管理员相关安全要求：

1. 管理员接口必须与普通分享接口分层。
2. `cookie` 或 `api key` 必须短时有效，并支持失效。
3. 所有密码都不能出现在公开分享 URL 中。
4. 运行时随机密码重启后会变化，客户端应允许重新认证。
5. 一旦拿到短期凭证，后续请求不得再默认重复发送明文密码。

### 5. 部署架构

实现路线：

1. 将后端服务和依赖写入 `Dockerfile`。
2. 使用 `docker-compose` 或等价方式组织：
   - API 服务
   - Nginx 反向代理
   - SQLite 数据卷与临时文件目录挂载
3. Nginx 负责：
   - 统一公开入口
   - 反向代理 API
   - 处理 `X-Accel-Redirect`
   - 承担受控静态文件发送
4. 部署到 Proxmox 中的轻量 LXC 或虚拟机。
5. 使用 FRP 暴露公网访问入口。
6. 配置固定域名：
   - API 域名
   - Web 播放页域名
   - 或统一域名下按路径区分
7. 增加基础运维能力：
   - 上传大小限制
   - 请求日志
   - 基础鉴权或签名，防止接口被滥用
   - 启动时输出运行时管理员凭证的受控展示方式

推荐部署边界：

- 单机即可运行
- 文件存储使用本机磁盘目录
- Nginx 与应用共享受保护的音频目录
- SQLite 文件与存储目录分别挂载卷，便于迁移和清理
- 没有必要首版就拆分为多服务

验收标准：

- 新上传内容可立即访问
- 过期内容会按预期清除
- 外网手机网络下可稳定上传和访问

### 6. 阶段二验收标准

满足以下条件即可视为后端首版完成：

- 能接收 Android 端上传的转码音频和元数据
- 能接收客户端指定的分享有效时长，并由服务端校验后生效
- App 安装后能生成持久 `client_install_id`，且后续请求均附带
- 能为每个上传生成内部 UUID 和对外 16 位分享路径
- 能自动生成指向前端的分享 URL，格式为 `base_url/<16位随机串>`
- 能通过 `/track/{share_code}` 提供前端渲染数据
- 能通过 `/stream/{share_code}` 配合 Nginx `X-Accel-Redirect` 稳定提供整文件下载
- 能让客户端按 `client_install_id` 查询自己分享的状态
- 能让客户端提前结束分享，并立即对外失效
- 后端启动时能随机生成认证密码
- 后端能通过密码自动判定普通权限和管理员权限
- 认证成功后能下发约 1 天有效的 `cookie` 或 `api key`
- 客户端启动时能自动申请短期凭证并复用
- 仅拥有管理员权限凭证的客户端可访问管理接口
- 管理员可查看已有音乐并执行管理动作
- 能在 24 小时后自动清理文件和记录
- 后端本身不承担转码、长期存储或复杂业务逻辑

## 阶段三：前端展示（Web Player）

前端职责是把分享结果变成一个足够美观、足够易用、但不强调下载能力的试听页面。

### 1. 前端技术栈与工程结构

前端统一采用 `Vue 3 + Vite`，以 SPA 形式交付。

建议技术栈：

- `Vue 3`
- `Vite`
- `Vue Router`
- `Pinia`
- `TypeScript`
- `Axios` 或原生 `fetch`

推荐目录结构：

- `src/app`：应用初始化、路由、全局样式
- `src/pages`：页面级组件
- `src/components`：可复用 UI 组件
- `src/stores`：Pinia 状态管理
- `src/services`：接口请求封装
- `src/types`：接口类型定义
- `src/utils`：时间格式化、复制文本、播放器辅助方法
- `src/assets`：静态资源和占位图

推荐首版文件拆分：

- `src/main.ts`
- `src/App.vue`
- `src/router/index.ts`
- `src/pages/TrackPage.vue`
- `src/pages/ExpiredPage.vue`
- `src/pages/NotFoundPage.vue`
- `src/components/TrackHero.vue`
- `src/components/AudioPlayer.vue`
- `src/components/MetadataCopyPanel.vue`
- `src/components/StatusBanner.vue`
- `src/stores/track.ts`
- `src/services/api.ts`

### 2. 页面路由与数据获取

实现目标：

- 用户打开分享链接后立即进入歌曲页
- 页面能稳定处理加载中、失效、异常三类状态
- 页面加载时将音频完整拉取到前端，减轻后端持续连接负担

实现路线：

1. 使用 `Vue Router` 定义核心路由：
   - `/:shareCode`：歌曲试听页
   - `/expired`：链接已失效页
   - `/:pathMatch(.*)*`：404 页
2. 在歌曲页加载时解析 URL 中的 16 位分享路径标识。
3. 调用后端 `GET /track/{share_code}` 拉取歌曲信息。
4. 根据接口结果更新页面状态：
   - `loading`
   - `ready`
   - `expired`
   - `error`
5. 当返回可播放资源时，前端应立即请求 `stream_url`，将音频文件完整下载到浏览器侧。
6. 下载完成后将响应内容转为 `Blob URL`，再交给隐藏的 `HTMLAudioElement`。
7. 当资源已过期或不存在时，跳转到专门状态页，而不是停留在空白播放器页面。

推荐状态流转：

1. 进入 `TrackPage`
2. 读取 `shareCode`
3. 请求歌曲信息
4. 请求整段音频文件
5. 成功则生成本地 `Blob URL` 并渲染播放器与封面
6. 过期则显示失效态
7. 请求异常则显示重试入口

建议接口类型：

- `uuid`
- `share_code`
- `title`
- `artist`
- `album`
- `duration_ms`
- `cover_url`
- `stream_url`
- `expires_at`
- `audio_mime`

### 3. 状态管理与业务分层

虽然页面不多，但播放器、请求状态、复制文本状态都值得拆分，避免把逻辑堆进单页组件。

推荐状态拆分：

- `track store`
  - 当前 `share_code`
  - 歌曲元数据
  - 加载状态
  - 错误信息
- `player state`
  - 是否播放
  - 当前进度
  - 总时长
  - 文件下载状态
  - 是否拖动中
  - 播放模式
  - 本地 `blob_url`
- `copy state`
  - 勾选的文本字段
  - 最近一次复制是否成功

推荐分层方式：

- `services/api.ts` 只负责接口请求
- `stores/track.ts` 负责获取与缓存轨道数据
- `AudioPlayer.vue` 负责音频控制、播放模式和进度同步
- `MetadataCopyPanel.vue` 负责文本拼装和剪贴板逻辑

### 4. UI / UX 设计与组件划分

实现目标：

- 视觉上接近主流流媒体分享卡片
- 页面在手机端和桌面端都具备完整观感
- 操作路径足够短，不引入“下载站”气质

视觉方向：

1. 页面背景使用封面图的模糊放大层，并叠加半透明暗色遮罩。
2. 中间主体采用单卡片布局，避免多栏信息过散。
3. 标题区突出歌曲名，艺术家和专辑作为次级信息呈现。
4. 页面底部弱化技术信息，只保留必要的失效时间提示。

推荐组件结构：

- `TrackPage.vue`
  - 负责页面装配和状态切换
- `TrackHero.vue`
  - 展示封面、标题、艺术家、专辑、失效时间
- `AudioPlayer.vue`
  - 自定义播放按钮、进度条、时间显示
- `MetadataCopyPanel.vue`
  - 文本项勾选、复制按钮、复制反馈
- `StatusBanner.vue`
  - 加载失败、过期、网络异常提示

播放器组件要求：

1. 不使用原生可见 `<audio controls>`。
2. 使用隐藏 `audio` 元素承载播放能力。
3. 音频源不是远程流地址，而是前端完整下载后生成的本地 `Blob URL`。
4. 页面中的播放器应为内嵌小型播放器，不跳转、不弹出独立播放页。
5. 自定义以下控件：
   - 播放 / 暂停按钮
   - 拖动进度条
   - 当前时间 / 总时长
   - 播放模式切换
6. 播放模式仅提供两种：
   - 播放完成停止
   - 单曲循环
7. 由于音频文件已经完整驻留在前端页面中：
   - 拖动进度条应直接在本地音频对象上完成
   - 暂停和恢复不应再次请求后端
8. 需要处理以下边界：
   - 音频完整下载前不可播放，或仅在下载完成后启用控件
   - 自动播放失败
   - 音频加载超时
   - iOS / Android 浏览器首次交互后才允许播放

文本复制面板要求：

1. 支持勾选以下字段：
   - 歌曲名
   - 艺术家
   - 专辑
2. 默认至少勾选歌曲名和艺术家。
3. 点击复制时将勾选内容按统一格式拼接，例如：
   - `Song - Artist`
   - `Song / Artist / Album`
4. 复制成功后提供轻量反馈，不打断播放流程。

### 5. 交互细节与异常场景

首版前端必须把以下交互补完整：

1. 加载态：
   - 页面首次进入先显示封面占位、骨架屏或简洁加载提示
   - 当音频正在完整下载时显示明确进度或“正在载入音频”
2. 失效态：
   - 明确提示“该分享链接已过期”
   - 不再展示可播放控件
3. 异常态：
   - 网络错误时提供重试按钮
   - 服务器异常时提示稍后重试
4. 无封面态：
   - 使用默认唱片或抽象占位图
5. 超长文本：
   - 歌曲名和专辑名支持省略
   - 同时保留完整 `title` 供悬停或长按查看
6. 播放模式交互：
   - 用户可在“播放完成停止”和“单曲循环”之间切换
   - 当前模式应有明确视觉状态

建议前端文案保持克制：

- 强调“试听”
- 强调“链接会过期”
- 不强调“下载”或“永久保存”

### 6. 样式与响应式策略

实现目标：

- 在手机浏览器中观感优先
- 在桌面端保持居中卡片化体验

推荐布局策略：

1. 移动端优先设计，宽度以单列为主。
2. 桌面端使用居中容器，限制最大宽度。
3. 封面比例固定为正方形，避免不同图片尺寸造成页面跳动。
4. 播放器和复制面板采用纵向堆叠，保证窄屏可读性。

样式建议：

- 使用 CSS 变量统一管理颜色、圆角、阴影、毛玻璃强度
- 背景和主体层级明显区分
- 避免全白背景或默认组件库观感

### 7. 前后端联调约定

为减少后续返工，前端需要提前和后端固定以下约定：

1. `GET /track/{share_code}` 的状态码语义：
   - `200`：可正常播放
   - `404`：资源不存在
   - `410`：资源已过期
2. 返回体字段命名全部固定为 snake_case 或 camelCase，避免混用。
3. `stream_url` 必须支持浏览器一次性完整下载音频文件。
4. `cover_url` 失效时前端可优雅降级，不阻塞主流程。
5. `expires_at` 需返回标准时间格式，前端统一换算为本地时区展示。

### 8. 阶段三验收标准

满足以下条件即可视为 Web UI 首版完成：

- 基于 `Vue 3 + Vite` 搭起可部署的 SPA
- 访问分享链接后能正确拉取并渲染歌曲数据
- 前端能在页面加载时完整拉取音频并生成本地播放源
- 自定义小型播放器支持暂停、拖拽进度条、播放完成停止、单曲循环
- 页面具备加载态、失效态、异常态
- 支持复制歌曲文本信息
- 手机端和桌面端都具备完整可用体验

### 9. 体验边界

这个项目的目标是“分享试听”，不是“云盘存储”。

因此前端需要明确体现：

- 链接是临时的
- 页面主打试听和信息展示
- 不强调文件下载入口
- 不提供文件管理、批量列表、历史库等超出 MVP 范围的能力

## 跨阶段公共任务

### 1. 协议先行

在移动端、后端、前端同时开工前，先固定一份最小协议文档，至少包括：

- 上传请求字段
- 分享响应字段
- 轨道信息 JSON 结构
- 过期字段格式
- 错误码约定

### 2. 隐私与安全

需要避免把用户本地真实目录结构直接暴露给前端或服务端日志。

建议：

- 客户端不要上传原始绝对路径
- 若确实需要追踪来源，上传内部哈希或脱敏值
- 服务端日志中不记录本地文件名全路径

### 3. 观测与排障

MVP 阶段至少保留以下日志：

- 收到的 Poweramp 广播摘要
- URI 解析成功 / 失败原因
- 转码耗时
- 上传耗时
- 服务端清理任务结果

## 推荐实施顺序

1. 先做 Android 端状态监听和 SAF 授权。
2. 再验证从 `path` 到 `documentUri` 的映射是否稳定。
3. 然后打通本地转码。
4. 再实现最小后端上传接口。
5. 最后补 Web 播放页和过期回收机制。

## 第一版 MVP 定义

满足以下条件即可视为第一版可用：

- 能从 Poweramp 获取当前曲目
- 能在 Android 10+ 设备上通过 SAF 找到真实文件
- 能通过 Tile 一键触发分享
- 能转码为浏览器可播放的 OGG
- 能上传并返回临时分享链接
- 链接打开后可看到封面、歌曲信息并试听
- 24 小时后链接自动失效且资源被清理
